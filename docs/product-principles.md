Product Documentation Rule

The project uses four primary planning documents, each with a distinct purpose. They should not overlap.

product-blueprint-v1.md

Purpose: Vision

Answers:

What experience are we trying to create?

Contains:

Product vision
User workflows
Product philosophy
Navigation
Long-term roadmap
Milestones
project-status.md

Purpose: Engineering Progress

Answers:

What has been built?

Contains:

Completed features
Current architecture status
Recently completed work
Current implementation state

It should never become a roadmap.

launch-checklist-v1.md

Purpose: Product Readiness

Answers:

Can we ship Version 1?

Contains:

Launch blockers
Progress percentages
Required features
Nice-to-have features
Beta readiness
Testing status
Release criteria

It should never become an engineering changelog.

next-priority.md

Purpose: Sprint Planning

Answers:

What should we build next?

Contains:

Immediate priorities
Ordered sprint recommendations
Short-term roadmap

Nothing more.

Decision Journal

Answers:

Why did we choose this approach?

Contains:

Product decisions
Tradeoffs
Rationale
Lessons learned
Guiding Principle

Each document should answer one question extremely well.

If information belongs in two documents, reconsider where it truly belongs.
